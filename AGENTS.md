# AGENTS.md — 跨模型工程代理主規範 V3（精簡版）

本文件適用於 Codex、Claude Code、Gemini CLI、OpenClaw、Copilot 類工程代理。目標：**低返工、可驗證、可回滾、少 token、少污染 diff**。

> 按需載入：個人環境讀 `AGENTS.personal.md`；工業自動化 / 機器視覺 / OCR / 相機 / PLC / 光源 / 100ms SLA 讀 `AGENTS.industrial-cv.md`。

## 0. 規範強度與優先級

### 0.1 強度定義
- **必須 / 必做**：不得跳過；無法執行時要回報原因。
- **建議**：預設採用；小型任務或不適用時可跳過。
- **可選**：依專案、工具與任務規模決定。
- **禁止**：不得執行；除非使用者明確要求且不違反安全原則。

### 0.2 文件優先級
1. 使用者本輪明確指令。
2. 目前工作目錄或子目錄中的代理規範：`AGENTS.md`、`CLAUDE.md`、`GEMINI.md`、`CODEX.md`、`COPILOT.md`、`OPENCLAW.md` 或等效文件。
3. 本文件。
4. README、CONTRIBUTING、既有程式碼風格與團隊慣例。

### 0.3 衝突處理
- 子目錄規範優先於根目錄規範，但不得覆蓋安全、誠實、不可逆操作確認。
- 使用者指令優先於文件規範，但不得違反安全原則。
- 若規範互相衝突且影響安全、資料、架構、部署或成本，**停止並回報**。

## 1. 核心哲學

1. **先理解，再動手**：不猜測，不隱瞞混淆，先釐清可影響方案的未知。
2. **極簡至上**：只做當前需求；禁止推測性功能、未要求的彈性與單次使用抽象。
3. **外科手術式修改**：只碰必要範圍；禁止順手重構、順手格式化、順手清理鄰近正常程式碼。
4. **證據驅動**：結論必須能由程式碼、文件、測試、指令輸出或 diff 支撐。
5. **安全與可回復優先**：不可逆、高風險、資料破壞、權限與敏感資訊操作必須先確認。
6. **少 token、少檔案、少噪音**：小任務不產生工作檔；中大型任務才維護 `.agent/*`。

## 2. 通用基礎規則

### 2.1 語言與編碼
- 對使用者回覆、任務紀錄、內部文件優先繁體中文。
- 程式碼命名、API、錯誤訊息、測試名稱依專案既有風格。
- 檔案一律使用 **UTF-8**。
- 寫檔、Shell/PowerShell 重導、外部程序日誌落盤必須明確指定 UTF-8。
  - Python：`open(path, "w", encoding="utf-8")`
  - PowerShell：`Out-File -Encoding utf8`

### 2.2 不確定性處理
- 低風險、可回復、局部細節：寫出假設後可繼續。
- 影響架構、資料、安全、部署、成本、硬體或 SLA：必須停止確認。
- 不得為了「看起來完成」而默默選擇高風險方案。

### 2.3 主動中止條件
遇到下列情況，必須停止並回報：
- 任務需求與安全原則直接衝突。
- 缺少不可推斷的關鍵資訊，且繼續會造成不可逆變更。
- 發現任務範圍與原理解有重大落差。
- 發現資料損毀、不一致，或繼續可能擴大損害。
- 工具結果不明確，且 diff / 狀態不符合預期。

回報格式：
- 中止原因：
- 已完成到哪：
- 目前狀態：可回滾 / 不確定 / 已變更
- 回滾方式：
- 建議下一步：

## 3. 任務分級與留痕

### 3.1 Fast-Track 判斷
| 規模 | 條件 | 路徑 | 留痕 |
|---|---|---|---|
| 小型 | ≤3 檔、無架構影響、無新依賴、無高風險 | 直接定位 → 修改 → 驗證 | 不必建 `.agent/*`，回覆列摘要與驗證 |
| 中型 | 4–10 檔，或介面 / 設定 / 資料格式變更 | 精簡 4 階段 | 建議 `.agent/workspace.md`，必要時 `.agent/verification.md` |
| 大型 | >10 檔、跨模組、新功能、重構、資料遷移 | 完整 4 階段 | 必須維護 `.agent/workspace.md` 與 `.agent/verification.md` |
| 高風險 | 刪除、DB、git push/reset、權限、部署、敏感資訊 | 先確認 | 必須留痕 |

### 3.2 兩個工作檔
- `.agent/workspace.md`：上下文、開放疑問、方案、任務清單、驗收條件、風險。
- `.agent/verification.md`：環境、操作留痕、驗證命令、結果、失敗、回滾、未驗證風險。

## 4. 工作流程

### 4.0 修改前檢查
- 先看 `git status --short`。若有非本任務造成的未提交變更，不得覆蓋或批量格式化，必須回報。
- 搜尋既有實作、相似案例、測試入口與依賴版本。

### 4.1 階段 0：理解
小型任務只需確認：
- 知道要改哪個檔案 / 位置。
- 知道如何驗證。

中大型任務需確認：輸入/輸出、介面契約、影響範圍、測試方式、主要風險。

### 4.2 階段 1：規劃
- 小型：1–2 句說明做法即可。
- 中型：若有多種合理做法，列取捨；若只有一種，直接說明。
- 大型：必須至少 2 個方案，含維護成本、風險、相容性、可測試性。
- 驗收條件必須可驗證，不接受「讓它正常」這類模糊描述。

### 4.3 階段 2：實作
- 使用最小可驗證增量。
- 一般邏輯變更單步 diff 建議 ≤100 行；超過需說明原因並優先拆分。測試、文件、生成檔不計入。
- 每一行變更必須能追溯至需求或必要修復。
- 不得修改不相關格式、註解、死碼；發現問題只在報告中提及。

### 4.4 階段 3：審查與交付
檢查：需求覆蓋、YAGNI、風格一致、回歸影響、測試覆蓋、文件同步、敏感資訊、可回滾性。

交付摘要包含：改了什麼、為何如此、如何驗證、已知限制、下一步。

### 4.5 回溯觸發
| 當前階段 | 觸發條件 | 回溯至 |
|---|---|---|
| 階段 1 | 關鍵資訊缺口導致方案不穩 | 階段 0 |
| 階段 2 | 發現需求理解錯誤 | 階段 0 或 1 |
| 階段 2 | 驗證連續失敗且根因不明 | 階段 1 |
| 階段 3 | 發現需求未覆蓋 | 階段 2 |

## 5. 工具、環境與路徑

### 5.1 工具策略
| 能力 | 優先 | 降級 |
|---|---|---|
| 搜尋 | `rg` / 語意檢索 | `grep` + 手動 Read |
| 精準編輯 | `apply_patch` | Edit / Write 後檢查 diff |
| 官方文件 | Context7 / 官方文件 | README / release note / 保守結論 |
| 最新資訊 | Exa / web | 標註待確認 |
| 任務追蹤 | task manager | `.agent/workspace.md` |

### 5.2 Python 探測優先序
1. 專案文件、CI、腳本或設定檔指定的 Python。
2. 環境變數或 `.env` 的 `AGENT_PYTHON_PATH`。
3. 專案 `.venv`：`./.venv/Scripts/python.exe`、`./.venv/bin/python`。
4. 系統：`python` → `python3` → `py -3`。
5. 仍不可用：停止 Python 執行驗證，改靜態審查，並在驗證摘要標註風險。

### 5.3 命令列安全
- 路徑一律用雙引號。
- 優先使用 `/` 作為分隔符。
- 不把 token、密碼、個資直接寫進命令列或 log。

## 6. 程式碼品質

- 遵循 SOLID、DRY、關注點分離、YAGNI、KISS。
- 優先標準庫與既有成熟依賴；避免無必要自研通用元件。
- 同型重複未達 3 次，不建立抽象層。
- 若 50 行可清楚完成卻寫成超過 150 行，必須重新審視是否過度工程；若保留需說明原因。
- 命名清楚，避免縮寫與隱語。
- 註解說明意圖、限制與決策原因，不重述語法。

### 6.1 TODO / 佔位
禁止無追蹤依據的 TODO、MVP、空函式或假實作。若分階段必須保留 TODO，格式需包含追蹤標識、原因與完成條件：

```python
# TODO(task-123): 等 API v2 穩定後補上重試邏輯；完成條件：v2 endpoint 文件確認。
```

## 7. 測試與驗證

### 7.1 必須新增或更新測試
- 修改公開介面：函式簽名、API、CLI、設定格式。
- 修復 bug：需有能重現該 bug 的測試或最小重現。
- 新增業務邏輯分支、錯誤處理、邊界值。
- 修改資料格式、驗證規則、共用函式、pipeline 主流程。

### 7.2 可只做冒煙驗證
- 純文件更新。
- 無邏輯變更的格式 / 命名調整。
- 小型設定值調整且已被既有測試覆蓋。

### 7.3 無法驗證
必須標註：未執行原因、靜態審查範圍、若可執行應跑的命令、剩餘風險。

## 8. 依賴管理

引入新依賴前必須檢查：
1. 標準庫或既有依賴是否足夠。
2. 維護狀態；長期未更新需標註風險。
3. 已知漏洞；可用時執行 `pip audit` / `npm audit` 或等效工具。
4. 授權相容性；GPL 類需確認。
5. 是否已在 `requirements.txt`、`pyproject.toml`、`package.json` 或 lockfile 中鎖定。

## 9. 高風險操作

執行前必須明確確認：
- 刪除 / 移動大量檔案、不可逆批次修改。
- `git commit`、`git push`、`git reset --hard`。
- 修改系統設定、環境變數、權限、CI/CD、部署設定。
- DB `DROP/ALTER`、資料遷移、大規模 update/delete。
- 上傳或外送可能含敏感資訊的 log、資料、影像。
- 全域安裝 / 卸載 / 升級核心依賴。

確認格式：操作類型、影響範圍、風險、回滾計畫、驗證方式、是否繼續。

若使用者回覆「不繼續」：停止操作，說明已完成狀態；若已有變更，列出回滾方式；不得自行改走其他高風險路線。

### 9.1 回滾計畫格式
```text
回滾指令：
  git revert <commit-hash>
回滾後處理：
  - 設定檔還原：
  - DB migration rollback：
驗證回滾：
  - 執行 smoke test：
```

## 10. Git 規範

### 10.1 Commit 格式
`<type>(<scope>): <繁中subject>`

常用 type：`feat`、`fix`、`refactor`、`docs`、`test`、`chore`、`perf`、`build`、`ci`。

Body 用於說明原因、取捨或不易從 subject 看出的變更。Footer 用於：
- `BREAKING CHANGE: <說明>`
- `Closes #42` / `Refs #42`

### 10.2 Pre-commit Audit
提交前必須檢查 staged 內容：
```bash
git status --short
git diff --cached --stat
git diff --cached
git diff --cached | grep -iE "(key|token|secret|password|api_key)"
```

不得夾帶：token、key、password、`.env`、私鑰、內部 IP/網域、個資、log、大型資料、快取、模型權重、非必要生成檔。若專案需納入小型測試 fixture，需確認用途與大小合理。

### 10.3 Push 保護
- 禁止直接 push 至 `main`、`master`、`release/*` 等受保護分支，除非使用者明確授權。
- push 前再次確認目標分支、commit 清單與風險。

- 修改公開 API、CLI、設定格式。
- 修改安裝 / 執行 / 部署流程。
- 新增環境變數。
- 改變資料格式、模型輸入 / 輸出、驗證規則。

## 12. 最終回覆格式

- 摘要：改了什麼。
- 驗證：跑了什麼命令 / 結果。
- 風險：未驗證或剩餘問題。
- 下一步：只列必要建議。
